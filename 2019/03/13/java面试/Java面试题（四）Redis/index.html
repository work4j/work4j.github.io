Java面试题汇总（四）Redis

Redis和Memecache有什么区别？

  • Redis支持的数据类型要比Memecache丰富
  • Redis支持数据磁盘持久化存储，Memecache不支持持久化
  • Redis支持主从，Memecache不支持
  • Redis支持分片，集群。Memecache不支持

为什么Redis能这么快？

Redis支持10万+的QPS，能这么快，因为以下几点：

  • 基于内存操作，执行效率高
  • 数据结构简单，对数据操作也简单
  • 采用的单线程，单线程也能处理高并发
  • 使用的多路I/O复用模型，非阻塞IO

Redis的使用场景都有哪些？

  1. 缓存
  2. 计数器
  3. 排行榜，去top n个数据
  4. pub、sub发布订阅构建实时消息系统
  5. 构建消息队列
  6. 精确的设置过期时间
  7. 各数据类型的应用场景：
    • String：缓存、限流、计数器、分布式锁、分布式Session
    • Hash：存储用户信息、用户主页访问量、组合查询
    • List：微博关注人时间轴列表、简单队列
    • Set：赞、踩、标签、好友关系
    • Zset：排行榜

如何从海量的key中查询出某一固定前缀的key？

  • keys xx*，但是数据量过多时，会造成卡顿，如果是线上环境，最好不要这样操作。
  • scan 游标位置 match xx* count 10，通过scan命令来查询。

如何通过redis来实现分布式锁？

  1. setnx key value：如果key不存在，则创建并赋值。成功返回1，否则返回0。配合expire key 秒数来解决key长期有效的问题。但是这样两个命令配合使用后，就不是原子操作了，推荐用第二种方式
  2. set key value EX 秒数 PX 毫秒数 NX|XX：NX是只有键不存在时，才对键进行操作。XX是只有键存在时，才对键进行操作。操作成功后返回ok，否则返回nil。

大量的key同时过期的注意事项？

大量的key同时过期的时候，集中清理时，可能会出现卡顿的情况。可以在批量设置key的时间时，加上随机数来解决。

如何使用Redis做异步队列？

  1. 使用List作为队列，rpush生产消息，lpop消费消息：缺点很明显，没有等待队列里有值就去消费了，可以sleep机制去调用lpop进行重试。
  2. 还是使用List作为队列，rpush生产消息，但是用blpop来消费消息，blpop key timeout秒数：阻塞直到队列里有消息或者超时。缺点是只能供一个消费者进行消费。
  3. pub/sub：主题订阅者模式。subscribe 主题，publist 主题 内容，缺点是消息的发布是无状态的，无法保证到达。

谈谈Redis的持久化？

  • RDB（快照）持久化：保存某个时间点的全量数据快照，redis默认开启的

    • 手动触发可以使用两种命令，save和gbsave，save命令会阻塞进程，gbsave会fork一个子进程进行创建rdb，不会阻塞redis服务器进程。
    • 自动触发的方式：
      1. 根据redis.conf配置里的save m n定时触发
      2. 主从复制时，主节点自动触发
      3. 执行debug reload
      4. 执行shutdown时且没有开启aof持久化

  • AOF持久化：记录下除了查询以外的操作指令，append到aof文件中。

    1. 开启：修改redis.conf文件中appendonly yes。
    2. 三种aof持久化方式：appendfsync always、everysec、no

  • RDB-AOF混合持久化：BGSAVE做镜像全量持久化，AOF做增量持久化。通过配置aof-use-rdb-preamble来开启。

    preamble

Redis高可用？

redis主从复制，在redis.conf里配置主节点。

解决主从同步主节点宕机问题，通过redis哨兵（sentinel），配置sentinel.conf文件，设置多个哨兵，监控各redis节点的状态，如果主节点宕机，自动进行主从切换。

Redis设置过期时间问题？

Redis可以通过set key value ex seconds或者expire key seconds设置时间，到期后会自动删除。比如手机验证码、session信息等，可以设置过期时间存在redis里面。

Redis清除过期key时，有两种方式：

  • 定期删除：redis默认每隔100ms就随机抽取一部分设置了过期时间的key，判断是否过期，过期则进行删除。之所以是随机抽取，是考虑到当redis里面有大量的key设置的有过期时间时，每100ms就对所有设置了过期时间的key进行遍历的话，会给cpu带来很大的负载。
  • 惰性删除：对于没有被定期删除的过期key，当查询到这些key的时候，会进行删除。

通过定期删除+惰性删除来清除过期的key，但是当定期删除漏掉了很多过期的key，但是有没有及时的去查，内存中就会有大量的无效key，导致redis的内存耗尽。针对这种情况，redis通过内存淘汰机制来应对。

什么是缓存穿透、缓存雪崩、缓存击穿？

  • 缓存穿透：查询大量不存在的key，缓存中没有，会查询数据库，而数据库里也没有，就不会写入缓存。这样大量的不存在的key查询，就会一直查询数据库，相当于没有了缓存一样。解决方式：

    1. 前端、后台做好数据格式校验，比如id不能小于0，邮箱格式错误等，直接将错误信息返回给客户端。
    2. 简单粗暴一些，缓存无效key，当数据库没有查询到数据时，也将这个空结果存放在缓存里，设置个比较短的过期时间。SET key value EX 60。
    3. 布隆过滤器：将所有可能存在的key放在布隆过滤器中，当查询key时，先经过布隆过滤器，判断是否存在，不存在的话直接返回请求参数错误信息给客户端。如果存在再进行下一步。

  • 缓存雪崩：当某个时间点大量的key过期时，大量的请求就会直接落在数据库上，导致数据库崩掉。解决方式：

    1. 将失效时间分散开：设置过期时间时，在原有及过期时间基础上增加一个随机值，这样大量key同时失效的几率就降低了。
    2. 加锁或者队列的方式保证缓存单线程写。

  • 缓存击穿：当一个key过期时，大量的请求访问这个key，导致大量的请求落在数据库上，导致数据库崩掉。

    ​ 常见的做法就是设置互斥锁。当一个key结果为空的时候，不是立即去数据库查询，而是先设置一个互斥锁，再去数据库查询key，查询到写入缓存后再删除锁。常用的命令：set key value EX 秒数 PX 毫秒数 NX|XX。
