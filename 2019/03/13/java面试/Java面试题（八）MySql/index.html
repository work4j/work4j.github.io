Java面试题汇总（八）MySQL

MyISAM和InnoDB区别？

MyISAM和InnoDB都是MySQL的存储引擎。

MyISAM是MySQL5.5之前默认的存储引擎，之后的版本InnoDB是默认存储引擎。

两者的对比：

  • 是否支持行级锁：InnoDB支持行级锁（默认）和表级锁，而MyISAM只支持表级锁。
  • 是否支持事务和崩溃后的安全恢复：InnoDB支持事务，具有事务、回滚、崩溃修复能力。而MyISAM强调的是性能，不支持事务，并且崩溃后无法安全恢复。
  • 是否支持外键：InnoDB支持，MyISAM不支持。
  • 是否支持MVCC（多版本并发控制）：InnoDB在RC和RR两个隔离级别下支持MVCC，MVCC可以用乐观锁和悲观锁实现，而MyISAM不支持。

说一说MySQL的索引？

MySQL的索引的数据结构主要有BTree索引和哈希索引。哈希索引的底层是哈希表，如果绝大多数需求是查询单条记录的时候，可以用哈希索引，查询性能更快，其他大部分场景使用BTree索引。

MySQL的BTree索引使用的是B+Tree，但对于两种存储引擎的实现是不同的：

  • MyISAM：B+Tree的叶子节点data域存放的是数据记录的地址，在索引检索的时候，先通过B+Tree的搜索算法搜索索引，找到key值后，取出data域里的值，并作为地址读取相应的数据记录，这也叫“非聚簇索引”、“非聚集索引”。MyISAM的索引文件和数据文件是分开的两个文件。

  • InnoDB：其数据文件本身就是索引文件，叶子节点data域保存了完整的数据记录，这个索引的key是数据表的主键，因此InnoDB的主键索引，也叫“聚簇索引”、“聚集索引”。其他键为辅助索引，辅助索引的叶子节点data域存储的是主键值，而不是地址。在通过主键索引搜索时，可以直接根据key所在的节点取出数据，用辅助索引时取出的是主键值，需要再走一遍主索引，才能取出数据，因此设计表时，不建议用过长的字段作为主键，也不建议用非单调的字段做主键，这样会造成主索引频繁分裂。

额外知识：InnoDB必有一个聚集索引：

  • 当一个主键被定义，该主键会被建立一个聚集索引。
  • 若主键没有定义，那么该表的第一个唯一非空索引会被座位聚集索引。
  • 若不满足以上条件，InnoDB内部会生成一个隐藏主键，建立聚集索引。
  • 之所以建立聚集索引，是因为辅助索引的存储的值是主键值，会再聚集索引上再次查找。

说一说MySQL的事务？

事务是逻辑上的一组操作，要么都执行，要不都不执行。

事务有四大特性：ACID

  • 原子性（Atomicity）： 事务是最小的执行单位，不允许分割。事务的原⼦性确保动作要么全部完成，要么完全不起作用。
  • ⼀致性（Consistency）： 执行事务前后，数据保持⼀致，多个事务对同⼀个数据读取的结果是相同的。
  • 隔离性（Isolation）： 并发访问数据库时，⼀个用户的事务不被其他事务所干扰，各并发事务之间数据库是独⽴的。
  • 持久性（Durability）： ⼀个事务被提交之后。它对数据库中数据的改变是持久的，即使数据库发生故障也不应该对其有任何影响。

并发事务带来哪些问题？

在应用程序中，会出现多个事务并发运行，彼此操作相同的数据完成各自的任务。在并发执行事务的情况下，会导致以下问题：

  • 脏读：一个事务A执行过程中，修改了一条数据，但是未提交，这是另一个事务B也读取了这条数据，这时读取的数据就是“脏数据”，产生脏读。
  • 丢失修改：一个事务A修改了一条数据，未提交前，另一个事务B也修改了该数据，事务A提交后，B事务也提交，此时事务A的修改被丢失。
  • 不可重复读：一个事务A多次读同一数据，读的过程中，另一个事务B对数据进行了修改，导致事务A多次读出来的数据不一致，称为不可重复读。
  • 幻读：一个事务A读取了几行数据，另一个事务B进行了插入或删除操作，导致事务A再次读取时发现多出几行或者少了几行数据，跟产生了幻觉一样，因此称为幻读。

事务的隔离级别有哪些，MySQL的默认隔离级别是什么?

SQL标准定义了4中隔离级别：

  1. READ-UNCOMMITTED（读取未提交）：最低的隔离级别，允许读取未提交的数据，可能会导致脏读、不可重复读、幻读。
  2. READ-COMMITTED（读取已提交）：允许读取并发事务已经提交的数据，可能导致不可重复读、幻读。
  3. REPEATABLE-READ（可重复读）：对同一字段的多次读取结果是一致的，除非数据被本身事务进行了修改。可以避免脏读、不可重复读，但幻读仍有可能发生。
  4. SERIALIZABLE（可串行化）：最高的隔离级别，完全符合ACID特性的隔离级别。所有事务依次逐个执行，这样事务之间就不会产生干扰，避免了脏读、不可重复读、幻读。

MySQL的InnoDB存储引擎默认的隔离级别是RR可重复读，但是MySQL在RR可重复读隔离级别下，使用了Next-Key Lock 锁算法，因此可以避免幻读的产⽣。

谈谈锁机制和InnoDB的锁算法?

  • 表级锁：会对当前操作的整个表进行加锁，实现简单，资源消耗少，加锁快，不会出现死锁。但是粒度大，触发锁冲突的几率高，并发度低。MyISAM和InnoDB都支持表级锁。
  • 行级锁：只针对当前操作的行加锁。行级锁能大大减少数据库操作的冲突，加锁粒度小，并发度高。但是加锁的消耗大，加锁慢，会出现死锁。只有InnoDB支持行级锁。

InnoDB的锁算法有三种：

  • Record Lock：单个行记录上的锁
  • Gap Lock：间隙锁，锁定一个范围，不包括记录本身
  • Next-key Lock：锁定一个范围，包括记录本身。Record +Gap

大表优化?

当数据库中某张表的数据量过大是，会影响CRUD操作，常见的优化措施：

  • 限定数据的范围：避免不带任何条件的去查询数据，比如查询订单历史的时候，控制在一个月的范围内。

  • 读写分离：主数据库负责写，从数据库负责读。

  • 垂直分区：大表拆小表，列的拆分。比如用户表既包括登录信息，又包括基本信息。就可以拆分成2张表，一张登录信息表，一张用户基本信息表。

    优点：表的列数据变小，简化结构，易于维护。查询时能够减少I/O次数。

    缺点：会产生主键的冗余列，而且还会引起join操作，也会让事务的管理变得复杂。

  • 水平分区：表结构不变，将数据进行分片。由于分片事务难解决，多节点的join性能差，逻辑复杂，所以尽量不要自己分片，因为拆分起来对逻辑、部署、运维都会带来复杂的工作。可以用中间件来做，如：MyCat。

解释⼀下什么是池化设计思想。什么是数据库连接池?为什么需要数据库连接池?

池化设计的实现案例有很多，比如java线程池，jdbc连接池，redis连接池等，池化设计是初始预设资源，减少频繁创建、销毁资源的开销。

数据库连接本质就是一个socket连接，数据库连接池就是初始预设一些数据库连接，并把这些连接缓存起来，当再次需要连接数据库时，就拿出来使用。在数据库连接池中，创建了连接后，就会放入到池中，并再次利用它，如果所有连接使用完了，会建立一个新的连接并添加到池中。连接池还能减少用户等待建立与数据库连接的时间。

sql执行顺序?

  1. 开启慢查询日志，定位到慢sql
  2. 通过explain，查看执行计划
  3. 分析执行计划，修改sql或者尽量让sql走索引
